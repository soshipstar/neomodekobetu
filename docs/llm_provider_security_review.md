## 1. 目的

care-bridge が利用する LLM 提供事業者 (OpenAI, Inc.) のセキュリティ体制およびデータ取扱いポリシーを定期的に評価し、本書に記録する。AISI ガイドの「モデルプロバイダーのセキュリティ体制 (データ暗号化、アクセス制御、第三者監査の実施状況、インシデント対応実績・情報開示姿勢) が評価されていること」 (表 3-7 ②) と「モデルプロバイダーのデータ取扱いポリシーが要件を満たしていること」 (表 3-6 ②) を満たす。

## 3. care-bridge 側の制御点

LLM 提供事業者のポリシーに依存しすぎないよう、care-bridge 側で実装している制御を以下に整理する。

| 制御点 | 実装 |
|---|---|
| 個人情報の最小化 | `AiIdentityMasker` で氏名・教室名を仮名化してから API 送信 |
| プロンプトインジェクション緩和 | `AiPromptSanitizer` でユーザー入力をデリミタ wrap |
| 出力後フィルタ | `AiGenerationService::moderate` で Moderation API による有害カテゴリ検出 |
| 入出力ログの非識別化 | `ai_generation_logs` には仮名化済みの内容を保管 |
| 改ざん防止 | `audit_logs` / `ai_generation_logs` のハッシュチェーン (R9) |
| アクセス制御 | Sanctum + `user_type` middleware + `ai_consent` middleware |
| 同意なし利用の遮断 | `RequireAiConsent` middleware で全 AI ルートを保護 |
| ZDR 未設定時の警告 | `OpenAiClientFactory::make` で `Log::warning` を出力 |

## 4. 評価サイクル

- **本書の見直し頻度**: 年 1 回、または以下のイベント発生時
  - LLM 提供事業者ポリシーの変更
  - SDK のメジャーバージョン更新
  - インシデント発生 (自社 / 提供事業者側)
  - 契約 (ZDR / DPA) の締結・更新
- **評価責任者**: セキュリティ担当 + 法務担当 + プロダクトオーナーの 3 者協議
- **記録**: 本書を更新し、`master_admin_audit_logs` に変更記録を残す

## 5. 将来の代替モデル評価

国内ホスト LLM や別プロバイダーへの切替検討時の評価マトリクス (将来の意思決定用):

| 候補 | 提供場所 | データ保持 | 国内法準拠 | 評価状況 |
|---|---|---|---|---|
| OpenAI gpt-5.4 系 | 米国 | ZDR 契約で制御可 | 個情法 28 条で要同意 | 利用中 |
| Azure OpenAI Japan East | 国内 | Azure コンプライアンス | 国内保存可 | 評価候補 |
| Anthropic Claude (Tokyo) | 国内 | Anthropic ポリシー | 国内保存可 | 評価候補 |
| 国内 OSS (Llama-jp 系) | オンプレ / 国内クラウド | 自社管理 | 国内保存可 | 評価候補 |

評価軸: 品質ベンチマーク (日本語 / 業務文書)、コスト、レイテンシ、ガバナンス、ベンダーリスク。
