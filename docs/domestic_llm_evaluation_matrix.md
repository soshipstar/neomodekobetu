# 国内ホスト / 代替 LLM プロバイダー評価マトリクス (care-bridge)

**バージョン**: v1.0
**最終更新**: 2026-05-17
**準拠**: AISI ヘルスケア AI セーフティ評価観点ガイド v1.0 / 4.3 (モデル選定), `docs/llm_provider_security_review.md`

---

## 1. 目的

要配慮個人情報の海外送信に関する規制動向 (個情法 28 条、3 省 2 ガイドライン) と、care-bridge の業務要件 (日本語 / 障害福祉ドメイン / 業務文書作成支援) を踏まえ、現行の OpenAI 利用と並行して検討すべき代替 LLM プロバイダーを定常評価する。

本マトリクスは将来の切替判断時の意思決定材料として、年 1 回程度の頻度で更新する。

---

## 2. 評価軸

| 軸 | 内容 |
|---|---|
| データホスト | 学習・推論のリージョン (国内 / 米国 / 欧州 等) |
| ZDR / DPA | Zero Data Retention 契約・データ処理契約の可否 |
| 国内法対応 | 個情法 28 条 / 3 省 2 ガイドライン / 次世代医療基盤法 への適合性 |
| 品質 (日本語) | 日本語業務文書生成品質 (体感ベース、care-bridge の typical prompt で評価) |
| 品質 (専門ドメイン) | 障害福祉 / 教育 / 医療隣接領域の語彙理解 |
| レイテンシ | API 応答時間 (95p) — 連絡帳生成 1000 token 程度 |
| コスト | 100 万トークンあたりの入力・出力単価 (相対値) |
| 既知の制限 | バイアス、ハルシネーション傾向、SaMD 関連制限 |
| ベンダーリスク | 提供事業者の事業継続性・規約変更頻度 |
| SDK / 統合容易性 | PHP / Laravel への組込容易性 |

---

## 3. 候補プロバイダー (2026-05 時点の評価草案)

凡例: ◎ 優れる / ○ 標準 / △ 課題あり / × 不適 / ? 未評価

### 3.1 OpenAI Direct (現行)

| 項目 | 評価 | 備考 |
|---|---|---|
| データホスト | 米国 | Enterprise + ZDR で制御可 |
| ZDR / DPA | ○ | 別途契約 (`OPENAI_ZDR=true` で運用切替可) |
| 国内法対応 | △ | 個情法 28 条で利用者同意取得が必須 (R3a で実装済) |
| 品質 (日本語) | ◎ | gpt-5.4 系で日本語業務文書品質は高い |
| 品質 (専門) | ○ | 障害福祉領域の固有用語に一部弱さ |
| レイテンシ | ○ | 日本〜米国西海岸 200ms〜 |
| コスト | ○ | mini モデルで運用 |
| 既知の制限 | △ | モデルカード参照必須 (`docs/llm_model_card_reference.md`) |
| ベンダーリスク | ○ | OpenAI の事業継続性は高いが、規約・モデル切替頻度が高い |
| SDK | ◎ | openai-php/client 公式 |

## 4. 切替判断基準

以下のいずれかが発生した場合、本マトリクスを根拠に切替検討を経営判断にかける。

1. **規制変更**: 個情法 28 条の運用変更、3 省 2 ガイドライン改定により、海外送信の同意要件が厳格化
2. **インシデント発生**: OpenAI 側でデータ漏えい / 規約違反 / 米国規制によるアクセス制限
3. **コスト構造変化**: 国内代替の単価が OpenAI Direct と同等以下に
4. **専門ドメイン要件**: 障害福祉 / 医療特化型 LLM (例: 国内 ELYZA-Med 系) の品質が業務要件を満たす
5. **顧客 (契約事業者) 要求**: 国内保存要件を契約条件として求められる

---

## 5. 切替時の手順 (`docs/llm_model_card_reference.md` §6 と整合)

1. 新プロバイダーのモデルカード / セキュリティ評価を `docs/llm_model_card_reference.md` / `docs/llm_provider_security_review.md` に追記
2. `OpenAiClientFactory` の抽象化を `LlmClientFactory` に拡張 (provider env で分岐)
3. ステージング環境で `OPENAI_PROVIDER=azure_japan_east` 等の env 切替
4. R12 テストスイート (`composer test:ai-safety`) 実行
5. 業務 typical prompt 20 件で品質比較 (ハルシネーション率 / 文体 / 専門用語)
6. プロダクトオーナー + 法務 + 医療専門家の 3 者承認
7. 規約 (`ai_usage_v1.md`) を更新してバージョン番号を上げ、ユーザーに再同意取得 (R3 経路)
8. 段階的本番反映 (フィーチャーフラグで一部教室から)
