# 設計案: 能力評価「到達マップ」(項目×学年帯で半年の成長を見える化)

作成: 2026-06-24 / ステータス: **レビュー待ち(承認後に着手)**

## 1. 目的

個別支援計画の目標が「できた/できない」の二値ではなく、**この児童が半年でどの力を
どこまで伸ばしたか**を、放デイの発達段階別(無学年)評価の趣旨で**縦断的に可視化**する。
同時に、現場の記入者が「広すぎる軸に当てはめにくい」負担を下げる。

## 2. 現状の課題

- マスターは**80軸 × 学年帯別到達目安246**を保持済(元表と1:1、欠落なし)。
- ただし採点は **項目ごとに0〜10を1つ**だけ持ち、可視化は**5領域レーダー**止まり。
  - 広い軸(例「思考力・問題解決」)が1スコア → **半年の細かい伸びが埋もれる**。
  - 学年帯(S1〜S6)は表示ラベルに留まり、**到達の前進が記録・可視化されない**。

→ 既にある **246の学年帯別到達目安を「評価の単位」に格上げ**すれば、新データ無しで
粒度が一気に上がる。

## 3. 中核アイデア — 到達マップ

児童ごとに「**項目 × 学年帯**」の格子を持ち、各マスの到達状況を記録・表示する。

| 能力(項目) | 小低 S1 | 小中 S2 | 小高 S3 | 中 S4 | 高前 S5 | 高後 S6 |
|---|---|---|---|---|---|---|
| 基本的生活習慣 | ✅到達 | ✅到達 | 🟡途上 | ・ | ・ | ・ |
| 数量・計算 | ✅到達 | 🟡途上 | ・ | ・ | ・ | ・ |
| 感情コントロール | ★般化 | ✅到達 | ✅到達 | 🟡途上 | ・ | ・ |

- 状態: `未着手` / `途上(1-7)` / `到達(8)` / `般化(9-10)`
- **半年の成長 = 期間内に前進したマス数・スコア増分**(児童全体／領域別／項目別)。
- ツール別の軸: DEV=成長段階S1〜S6 / ADV=到達水準L1〜L4 / WRK・UNV=時期P1〜P2。
  非高校生はDEVマップのみ(先の出し分け仕様を踏襲)。

## 4. データモデル(新テーブル不要)

既存 `ability_scores`(student_id, item_id, **axis_id**, score 0-10, evaluated_on, 履歴追記)
を **「項目×学年帯」単位で運用**する。

- 現状: 採点は `item_id` のみで集約 → 1項目1スコア(axis_idはラベル)。
- 変更: 採点を **`(item_id, axis_id)` 単位**で集約 → 学年帯ごとにスコア履歴を持つ。
- 到達マップ = 各 `(item, 学年帯)` の**最新スコア**から導出(到達=score≥しきい値)。
- 半年の成長 = `evaluated_on` 履歴で「6か月前時点の到達」と「現在の到達」を比較
  (append-onlyのため**スナップショット表は不要**)。

> 既存本番スコアは現行axisのまま履歴として残る。移行は破壊的変更なし(前進的)。

## 5. 到達判定・段階進行ロジック

- **今取り組む学年帯** = その項目で**まだ到達していない最も低い学年帯**(無ければS1から)。
  - 観察記録は「今取り組む学年帯」で保存(現行の固定S1を、到達で前進する方式へ)。
- **到達(achieved)** = 当該学年帯の最新スコアが **しきい値以上**。
  - 既定しきい値 = **8(一人で安定してできる=慣れた場面で9割)** 【※要確認】
  - 9-10 は `般化/定着` として上位表示(任意)。
- 到達したら次の学年帯へ前進(設問・到達目安が1段上がる)。
- 採点は既存ルールエンジン(支援量→成功率→般化)を**そのまま**学年帯別に適用。AIは採点に使わない。

## 6. 可視化

- スタッフ画面: 到達マップ(項目×学年帯グリッド、色分け)＋ **期間指定の成長サマリ**
  (例「直近6か月: 到達マス +7 / 般化 +2」、領域別の前進、伸びた項目TOP)。
- 既存の5領域レーダー(P4)は併存。レーダーは「現在地」、マップは「到達の広がりと前進」。
- 別添PDF(P4)に到達マップと成長サマリを追加(保護者・計画用)。

## 7. 現場運用(記入負担)

- 記入は**従来どおり1日1問**(児童ごと)。マップは日々の観察で**自然に埋まる**。
- 「どの項目を出すか」は **AIが今日の活動に合わせて選定**(別途設計済の案)を併用すると、
  記入者が広い軸を探す負担が下がる ←「つける側が困る」への対処。
- 数件未満の項目は採点保留(現行ルール踏襲)。マップ上は「途上」または「未着手」。

## 7-2. P-B 詳細設計(到達マップの可視化) ← 今回の承認対象

### (1) セル状態の定義(項目×学年帯ごと)
各 `(項目, 学年帯)` の**最新スコア**で1マスの状態を決める:

| 状態 | 条件(最新スコア) | 表示 |
|---|---|---|
| 未着手 `not_started` | スコア無し | ・(灰) |
| 途上 `in_progress` | 1〜7 | 🟡(黄) |
| 到達 `achieved` | 8 | ✅(緑) |
| 般化 `generalized` | 9〜10 | ★(濃緑) |

- `current_axis` = まだ到達していない最も低い学年帯(=今取り組む段階)
- `reached_axis` = 到達済みの最高学年帯

### (2) API
`GET /api/staff/students/{student}/ability/progress-map?since=YYYY-MM-DD&as_of=YYYY-MM-DD`
- 認可: 既存の能力評価ガード(自教室/越境403)、トグルOFFは409。
- `since` 既定 = as_of の6か月前、`as_of` 既定 = 今日。
- 応答(要点):
  - `tools[]`: ツールごとに `axes[]`(列=学年帯)＋ `domains[].items[].cells[]`(各マスの状態・スコア・到達日)。非高校生はDEVのみ。
  - `growth`: 期間内の成長サマリ
    - `achieved_delta`: 期間内に**新たに到達**したマス数
    - `score_gain_total`: 期間内のスコア増分合計
    - `by_domain[]`: 領域別の到達増・スコア増
    - `top_items[]`: 伸びた項目(例「基本的生活習慣: S1途上 → S2到達」)

### (3) 成長の算出(履歴ベース・スナップショット不要)
`ability_scores` は追記履歴なので、各 `(項目,学年帯)` について
**「`since` 時点の最新スコア」と「`as_of` 時点の最新スコア」**を比較して算出する。
- 新規到達 = `since` 時点は未到達(<8) かつ `as_of` 時点は到達(≥8) のマス。
- 新サービス `AbilityProgressMapService::forStudent(student, since, asOf)`。

### (4) フロント
- 新コンポーネント `AbilityProgressMap`(項目×学年帯のグリッド＝ヒートマップ、領域で折りたたみ)。
- 上部に**成長ヘッダ**(「直近6か月: 到達 +7 / 般化 +2 / 伸びた項目…」)＋期間切替。
- 設置: 個別支援計画画面の `AbilitySummaryView` 隣(レーダー=現在地 / マップ=広がりと前進)。
  児童詳細にも導線可。トグルOFF(409)時は自己非表示(既存コンポーネントと同方針)。

### (5) テスト
- API: マップが項目×学年帯の状態を返す / 成長(到達増・スコア増)が期間で正しい /
  越境403 / トグルOFF=409 / データ無し時の空表現。
- フロント: `tsc --noEmit`。

### (6) 進め方(P-B 内の分割)
- **P-B1 (api)**: `AbilityProgressMapService` ＋ エンドポイント ＋ テスト。
- **P-B2 (screen)**: グリッドUI ＋ 成長ヘッダ ＋ 設置。

## 7-3. P-C 詳細設計(段階別の具体設問をAIで用意) ← 今回の承認対象

### 背景(現場フィードバック 2026-06-25)
到達目安テキストをそのまま「設問」にすると、指導員が「何を観察して答えればよいか」が
分かりにくく、支援コード/結果/行動の入力もタイミングが合わず**答えられず困惑**している。
→ 各(項目×段階)に**具体的で答えられる問い**を用意し、入力は「該当度」に絞る(P-D)。

### (1) 生成物(項目×段階ごと)
まず**療育(DEV 5領域)× S1〜S6 = 150問**を AI で生成する。各マスに:
- `question`: 指導員が「どれくらいできているか」で答えられる**具体的な問い**
  (例 DEV-1-1×S1: 「声かけや見守りがあれば、手洗い・歯磨き・身だしなみを自分で行えていますか?」)
- `hint`: 観察の手がかり(具体例。「食事前後・トイレ後の手洗い、朝の整容 など」)
- 到達目安(`benchmark`)は**参照として残す**(問いの根拠)。

### (2) 生成方法
- artisan コマンド `ability:generate-stage-questions --tool=DEV [--force]` で 150 マスを
  バッチ生成。入力=領域/項目名/定義/判断の観点/段階名/その段階の到達目安。
- モデルは既存 `AiGenerationService` と同じ(gpt-5.4)。**児童PIIは含まない**(マスターのみ)が、
  A005ガード順守のため PiiMasker 経由(実質no-op)で送る。出力はJSON {question, hint}。
- 冪等: 既存はスキップ、`--force` で再生成。生成結果は確認用JSONにも書き出す。

### (3) 保存
- 新テーブル `ability_stage_questions`(item_id, axis_id, question, hint, model,
  generated_at, reviewed_at, is_active)。一意 (item_id, axis_id)。
- レビュー運用: 生成直後から日々の出題に使う(`is_active=true`)。後から編集・再生成可。
  生成物は人がJSONで確認・修正できる(他のマスター同様)。

### (4) 日々の出題への反映
- `AbilityQuestionService::buildQuestion` は、現在の学年帯に対応する
  `ability_stage_questions.question / hint` を返す(無ければ到達目安にフォールバック)。
- これで「S1の問い→(到達したら)S2の問い…」と**段階的に具体設問でヒアリング**できる。

### (5) テスト
- コマンド: AIクライアントをモックし、150マスに question/hint を保存(冪等)。
- buildQuestion: stage_question があればそれを返す/無ければ到達目安にフォールバック。

### (6) 進め方(P-C 内の分割)
- **P-C1 (schema)**: `ability_stage_questions` テーブル＋モデル。
- **P-C2 (logic)**: 生成コマンド(AIバッチ・DEV150)＋確認用JSON出力＋テスト。
- **P-C3 (logic)**: buildQuestion を stage_question 優先に(フォールバック付き)＋テスト。

> 後続: **P-D**=入力を「該当度」スケールに簡素化(支援コード→程度・コメント任意)、
> **P-E**=連絡帳「統合」時に各欄の下へ該当設問を表示。いずれも P-C の上で設計する。

## 8. フェーズ計画(各フェーズ=1コミット/デプロイ都度承認)

- **P-A (logic)**: 採点を項目×学年帯に / 観察を到達進行の学年帯で保存 / 到達導出。
  - 影響: `AbilityScoringService`(集約キー) `AbilityObservationController::store`(axis)
    `AbilityToolScope/AbilityQuestionService`(進行段階)。新規マイグレーションは原則不要
    (必要なら `ability_scores(student_id,item_id,axis_id)` の索引追加のみ)。
  - テスト更新: `AbilityScoringEngineTest`(学年帯別集計) `AbilityObservationFlowTest`。
- **P-B (api+screen)**: 到達マップAPI(項目×学年帯の到達・日付)＋ グリッドUI ＋ 成長サマリ。
  - 影響: 新コントローラ/サービス(例 `AbilityProgressMapService`)、フロント新コンポーネント。
- **P-C (screen, 任意)**: 別添PDFに到達マップ＋成長。必要なら AI設問選定の併用。

## 9. 影響範囲・リスク

- 稼働中のP3採点エンジンの**集約単位変更**(項目→項目×学年帯)。P4サマリ/別添への波及は
  「最新スコア=現在取り組む学年帯のスコア」で吸収。既存テスト更新が必要。
- 本番既存スコアは履歴として保持(前進的移行・破壊なし)。
- しきい値8の妥当性は現場感覚に依存 → **要確認**(7や9も選択可)。

## 10. テスト方針

- P-A: 学年帯別に到達/未到達/前進が正しく算出される(pgvectorで回帰)。
- P-B: マップAPIが項目×学年帯の到達と6か月成長を返す/越境403/トグルOFF=409。
- 既存能力評価スイート(24件)を緑に保つ。

## 11. 承認前に確認したい決定事項

1. 到達しきい値 = **8(安定して一人で)** で良いか(7=ほぼ毎回 / 9=般化 も可)。
2. マップの状態区分 = `未/途上/到達/般化` の4段階で良いか。
3. 進め方 = **P-A から段階的に実装→各フェーズ承認→デプロイ** で良いか。
4. AI設問選定(今日の活動に合わせる)は P-C で併用するか、先に独立で入れるか。
