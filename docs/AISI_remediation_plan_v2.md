## 1. care-bridge のスコープ該当性 (公式原文照合)

### 表 1-1 「本ガイドのスコープ」(公式 p.6) との対比

| 軸 | ガイドの対象範囲 | care-bridge | 該当 |
|---|---|---|---|
| **対象者** | AI 提供者 | care-bridge 運営会社 = LLM API 経由でプロダクト提供 | ✓ |
| **対象プロダクト** | 非医療機器プログラム (Non-SaMD) | 業務記録 / 連絡帳 / 個別支援計画支援。薬機法のプログラム医療機器に非該当 | ✓ |
| **対象生成 AI 種類** | テキスト生成 AI (LLM) | OpenAI `gpt-5.4-mini-2026-03-17` 経由 | ✓ |

### 表 1-2 「想定ユースケース」(公式 p.6) との対比

| カテゴリ | ガイドの例示 | care-bridge の該当機能 |
|---|---|---|
| **BtoB (医療従事者向け)** | 文書作成支援、情報検索・要約、患者説明資料の作成支援 | **連絡帳統合文 AI 生成 (`generateIntegrated`)**, **個別支援計画 AI 生成 (`generateAi`)**, **モニタリング報告 AI 生成**, **教室だより AI 生成 (`generateNewsletter`)** |
| BtoC (生活者向け) | 健康相談チャットボット、セルフケア支援、メンタルヘルスケアアプリ | 該当なし (保護者・利用者は AI を直接呼ばない) |
| その他 | 製薬企業向け情報提供支援 | 該当なし |

→ care-bridge は **BtoB 文書作成支援に分類される Non-SaMD LLM プロダクト** として、本ガイドの完全な対象範囲内。

### 1.2.3 (公式 p.7) の AI プロダクト定義
> 「AI モデル単体ではなく、**入力処理、モデル推論、出力制御、データベース、ユーザーインターフェース、運用基盤、利用規約・免責事項等を含む、エンドユーザーに提供される製品・サービス全体**を指す。」

→ care-bridge の評価対象は backend 全体 + frontend + 運用基盤 + (現在不存在の) 利用規約・免責事項を含む。

## 5. 改修プラン v2 (R1〜R13)

### v1 と継続する 8 件 (R1〜R8) は維持。以下は v2 追加・修正分。

### R1 プロンプトインジェクション対策 (公式 3.2.6 表 3-7 ③, 4.2.5(2))
**v2 で追加**: 公式に "プロンプトリーキング" と "間接プロンプトインジェクション" も明示されているため、`AiPromptSanitizer::detectLeakage()` の対象キーワードに **system プロンプト全文の reconstruction を疑うパターン** (例: モデル名直書き、API キー patterns) も追加。さらに `wrap()` のデリミタを単純な `<<<USER_INPUT>>>` ではなく、毎呼出ごとに **8 byte ランダム接尾辞付きデリミタ** (例: `<<<UI_a3f9b2c1>>>`) にすることで、ユーザー入力からデリミタ自身を予測・偽装する攻撃を防止。

### R2 仮名化レイヤ (公式 4.2.5(1) 個人情報の最小化, 3.2.5 表 3-6 ③)
**v2 で追加**: 仮名化対象を 3 種に拡大:
- **直接識別子** (実名, username, email) → 必須仮名化
- **準識別子** (生年月日 → 年齢/学年帯に丸める、住所 → 市区町村まで)
- **要配慮個人情報** (障害種別 → 大分類のみ送信、診断名は送信せず)

### R3 規約 + 同意 (公式 4.2.4 全法令への適合, 4.2.5(1)(4))
**v2 で追加**:
- 規約 `ai_usage_v1.md` に **3 省 2 ガイドライン適合確認** セクションを追加
- 規約 `ai_usage_v1.md` に **薬機法 SaMD 非該当の判断書** (Non-SaMD として位置付けた根拠) を添付
- 同意撤回時の **既存 AI 生成データの取扱い** (撤回後も既存出力は維持、新規生成のみ停止) を明記
- 個情法 28 条 (外国にある第三者への提供) として **米国 OpenAI** への提供を明示的に同意項目化

### R4 ハルシネーション抑制 + 医療免責 (公式 3.2.2 表 3-3 ③, 3.2.7 表 3-8 ②③)
**v2 で追加**:
- `AiPromptDirectives::systemBase()` に **「知識の境界において『回答不能』『不明』『追加情報が必要』『情報なし』と明示する」** を必須化 (公式 3.2.2(1) 直接引用)
- **Function Calling / response_format** の活用拡大: 全 AI 呼出を JSON schema 厳格化し、`uncertainty_flag` / `confidence_level` / `unable_to_answer` フィールドを追加して構造化された不確実性表示を実装。care-bridge は既に `response_format: json_object` を 4 箇所で使用しており、これを全 18 箇所に展開。

### R5 HITL ポリシー + AI 関与表示 (公式 4.2.5(3) HITL, 3.2.7 表 3-8 ③)
**v2 で再構成**:
- `IntegratedNote` の `ai_assisted` (v1 案) に加え:
  - `ai_review_status` (`pending`, `reviewed`, `modified`, `rejected`) を追加
  - `ai_reviewed_by` (user_id), `ai_reviewed_at`
- **HITL ポリシードキュメント** `docs/HITL_policy.md` を新設:
  - 各 AI 機能ごとに「介在レベル」(最終判断 / 監視介入 / 完全自動) を明示
  - 連絡帳 / 個別支援計画 / モニタリング = **すべて「人間が最終判断」レベル**
  - 教室だより = **「監視・介入可能」レベル**
  - 「**確認疲れ**」対策として、AI 出力の diff highlight 表示と、変更履歴の自動記録

### R6 OpenAI ZDR + DPA + 国内代替検討 (公式 3.2.5 表 3-6 ②, 4.2.5(1))
**v2 で追加**:
- ZDR に加え **DPA (Data Processing Agreement)** の締結確認を必須化
- `services.php` に `'data_processing_agreement_url' => env('OPENAI_DPA_URL')` を追加し、運用部門で URL 管理
- **国内代替** (Azure OpenAI Japan East / Anthropic Tokyo / 内製) の評価マトリクスを `docs/llm_provider_evaluation.md` に作成

### R7 Moderation API (公式 3.2.1 表 3-2 ③ 多層防御)
v1 から大きな変更なし。**出力層の多層防御** の一環として位置付け。

### R8 ログ purge (公式 3.2.5 表 3-6 ⑤, 3.2.10 表 3-11 ⑤)
**v2 で追加**:
- `audit_logs` / `ai_generation_logs` 各テーブルに **保持期間 (法令準拠で 5 年案)** を明文化
- **「変更管理と継続モニタリング」** (公式 3.2.10 表 3-11 ⑤) として、モデル更新時の全レコードへの記録を強化

### R9 監査ログの改ざん防止 (新規 — 公式 3.2.6 表 3-7 ⑤, 3.2.10 表 3-11 ③④)
公式: 「**改ざん不可能な監査ログ**」「**生成物の作成・修正・承認の履歴が保持され、改ざん防止が実装されている**」

**設計**:
- 既存 `audit_logs` / `ai_generation_logs` に **`row_hash` (sha256) と `prev_row_hash` カラム追加** → ハッシュチェーン化
- マイグレーション + DB トリガで自動計算
- 検証用 Artisan コマンド `audit-logs:verify-chain` で全チェーンの整合性確認
- (オプション) **AWS S3 Object Lock** または **CloudWatch Logs に WORM 化** して外部不変ストレージへ二重保管

**カテゴリ**: schema + logic (1 機能セット)

### R10 自傷念慮 / 緊急症状検出 + 相談窓口誘導 (新規 — 公式 3.2.1 表 3-2 ③)
公式: 「**高リスク文脈 (自傷念慮、緊急症状等) では安全優先モードへの動的切替および相談窓口への誘導**」

**care-bridge での該当場面**:
- 連絡帳の 5 領域所見テキスト (`notes`) に「死にたい」「リストカット」「殴られた」等が含まれる場合
- 個別支援計画の `student_wish` に同様の表現がある場合
- ヒヤリハット検出 (`detectHiyariHattoCandidate`) は既存だが「AI による予測」のみ

**設計**:
- `backend/app/Services/AiSafetyTriage.php` 新規
  - `containsHighRiskContent(string $text): array` で危険語句検出 (キーワード + sentence embedding 類似度)
  - 検出時の動作:
    1. AI 生成は **継続するが**、出力の冒頭に「**※ 緊急時相談窓口: いのちの SOS 0120-279-338 / 児童相談所 189**」を強制挿入
    2. master_admin_audit_logs に **アラート記録** (担当責任者に通知)
- 連絡帳 / 個別支援計画 / アセスメント の入力時点でも警告バナー表示

**カテゴリ**: schema + logic

### R11 モデルカード / システムカード参照 + サプライチェーン文書化 (新規 — 公式 3.2.10 表 3-11 ②, 4.2.5(2))
公式: 「**モデルカード・システムカードが公開されており、モデルの既知の制限事項・リスクが開示されていること**」「**モデルプロバイダーのセキュリティ体制 (データ暗号化、アクセス制御、第三者監査の実施状況、インシデント対応実績・情報開示姿勢) が評価されている**」

**設計**:
- `docs/llm_model_card_reference.md` 新規 — OpenAI gpt-5.4 のモデルカード URL / 既知の制限事項 / バイアス情報を保管
- `docs/llm_provider_security_review.md` 新規 — OpenAI のセキュリティ体制 (SOC 2 Type II, ISO 27001 等) を文書化
- 定期更新サイクル (年 1 回) を運用ポリシーに組み込む

## 10. 結論

公式 PDF の完読により、v1 で挙げた課題は方向性として正しかったが、以下の点で **より厳格な要件** が確認された:

- **改ざん不可能な監査ログ** (V6 / V10): 現状の `audit_logs` は Eloquent から update 可能で不適合。R9 で対応。
- **多層防御 (V1) と相談窓口誘導**: 連絡帳 AI 文中の自傷念慮・緊急症状検出と動的安全モード切替が必要。R10 で対応。
- **モデルカード参照とサプライチェーン評価** (V2 / V10): OpenAI のモデルカードと SOC 2 / ISO 27001 等を文書化する責務。R11 で対応。
- **Function Calling 等による出力制御** (V7): 不確実性表示を構造化する必要。R4 で対応。
- **HITL の明文化** (4.2.5(3)): 「人間が最終判断」の介在レベルを文書化する必要。R5 で対応。

これらを含む R1〜R12 (12 件) を完了すれば、care-bridge は AISI ガイド v1.0 の **実装側で達成可能な範囲で完全準拠** に到達できる見込み。
