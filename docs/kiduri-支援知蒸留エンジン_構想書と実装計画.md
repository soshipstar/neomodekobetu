# きづり 支援知蒸留エンジン — 構想書と実装計画 (v0.1)

出典: `きづり 支援知蒸留エンジン構想書.pdf`(報告者提供)。本書はその内容を仕様として取り込み、
既存実装(AI学習基盤 S0〜S5、本番稼働中)との対応・未実装ギャップ・実装フェーズを整理する。
AI学習基盤の上位ビジョンであり、企画書 `docs/kiduri-AI学習基盤_企画書.md` を拡張する。

---

## 1. 構想(原典の要約)

きづりは単なる相談サービス/記録システムではない。全国の放課後等デイサービスの
**支援者の記録・保護者の相談・利用者本人の声**を蓄積し、AIで構造化・蒸留して
**再利用可能な「支援知」**へ変換する知識基盤を目指す。

- 目的はAIチャットボットの構築ではなく、**「支援者の思考を育てながら、全国の支援実践を知識資産化する」**こと。
- 「放デイ版ChatGPT」ではない。**全国の支援実践を蓄積・蒸留する知識基盤**である。AIは回答する存在ではなく、
  支援者の思考を育て、支援知を構造化し、未来へ継承する存在。

### 1.1 基本思想: 事実記録 → 支援知記録
単なる事実(「今日は落ち着いて過ごせた」)ではなく、**因果仮説**まで蓄積する:
`状況 → 出来事 → 支援 → 結果 → 仮説`(「なぜその結果が起きたのか」)。

### 1.2 データ階層(4レベル)
- **L1 生データ**: 「今日は落ち着いていた」「活動に参加できた」「家で癇癪が増えた」
- **L2 AI構造化**: AIが `事実 / 支援 / 結果 / 仮説 / タグ` を抽出。
  例) 事実=工作活動への参加拒否 / 支援=個室で雑談 / 結果=30分後に自発参加 / 仮説=集団環境への負荷
- **L3 支援知**(個別ケースを超えた知見): 対象=中学生・ASD傾向・新学期後 / 有効支援=見通し提示・役割形成 / 注意=活動参加を強要しない
- **L4 原理**(複数ケースから抽出): 見通し形成 / 自己決定 / 安心基地形成 / 役割形成 / 感覚調整

### 1.3 AI記録支援機能(Socratic)
**AIが記録を書くのではない。AIが支援者へ問いを返す。**
- 入力「今日は落ち着いていた」→ AI「どの場面で？普段と何が違いましたか？どのような支援を？何が影響した?」
- **仮説生成**: AIが因果仮説を提案。例)考えられる要因①見通し提示②好きな活動③睡眠状態④人間関係の安定 → 該当は？

### 1.4 支援者成長モデル(Lv1〜4、AI支援が段階で変わる)
| Lv | 特徴 | AI支援 | 目的 |
|---|---|---|---|
| 1 新人 | 事実のみ記録 | 多数の質問・記録補助 | 観察力育成 |
| 2 初級 | 状況把握可能 | 仮説記述の促進 | 因果思考育成 |
| 3 中堅 | 仮説記述可能 | 記録添削(「因果関係の記述が不足」) | 支援設計力向上 |
| 4 ベテラン | 支援設計可能 | 類似事例・代替仮説提示(「感覚過敏要因の可能性も」) | 専門性深化 |

### 1.5 学習ループ(最重要)
`AI仮説 → 支援者修正 → 修正内容保存 → 支援知DB更新 → AI改善`。
**重要なのは完成記録ではなく「AIの推論 → 人間の修正」を保存すること。**

### 1.6 支援知データベース構造
`Case`(年齢/学年/特性/状況)・`Observation`(観察)・`Intervention`(支援)・`Outcome`(結果)・
`Hypothesis`(仮説)・`Evaluation`(本人/保護者/支援者の3評価)・`Knowledge`(蒸留された知見)。

### 1.7 将来構想(横断検索・根拠提示)
全国のケースから「どんな状況で・どんな支援を行い・どんな結果になったか」を検索可能に。
- 従来「ASDには見通し提示が有効です」→ 未来「新学期後・中学生・登校負荷増加・集団活動拒否では、
  **類似ケース842件中63%で見通し提示が有効**でした」(条件付き・根拠付きの提示)。

## 2. 既存実装(S0〜S5)との対応 — 学習ループは実現済み

構想1.5の学習ループ「AIの推論→人間の修正を保存→改善」は、現行 AI学習基盤が**すでに実現**している:

| 構想 | 現行実装(本番稼働) |
|---|---|
| AIの推論→人間の修正を保存 | S2/S3 配管: `ai_generation_events`(AI生成=before)+`ai_revision_events`(人間最終=after、暗号化)を4文書で蓄積 |
| 修正内容保存→改善(AIが学習) | S5 自己改善ループ: `WritingProfileService` が確定記述を生成プロンプトへ注入(4文書) |
| 同意・プライバシー | S1 同意基盤(AND・fail-closed)+ PII3層 + マスク + 構造化PIIスクラブ |
| 多次元の傾向把握 | S4 次元スナップショット(対象/作者/施設/成長段階)+ S4c Layer2集計 + S4d 管理者レポート |
| Evaluation(本人/客観) | mynameis主観(本人)× 能力評価(客観)を併記(能力評価システム連携) |
| タグ/分類 | S4 実施プログラム分類(5領域×種別、ルール+人手) |

→ **学習ループの「土台」は完成**。蒸留エンジンが追加で要求するのは、下記**未実装**部分。

## 3. 未実装ギャップ(蒸留エンジン固有)

| # | 構想要素 | 現状 | 不足 |
|---|---|---|---|
| G1 | **L2 構造化**(事実/支援/結果/仮説/タグの抽出) | 修正の before/after は持つが、因果フィールドに**分解していない** | 記録→`Observation/Intervention/Outcome/Hypothesis`への構造化抽出 |
| G2 | **AI記録支援(問い返し)** | 生成は「下書きを書く」方式 | 「問いを返す」Socratic対話・**仮説候補提示**のUI/API |
| G3 | **支援者成長モデル(Lv1〜4)** | 作者別傾向の集計はある | Lvの判定 + Lvに応じたAI支援の出し分け |
| G4 | **支援知DB(Case/.../Knowledge)** | event系テーブルのみ | L3支援知・L4原理を蓄える**蒸留テーブル**と蒸留処理 |
| G5 | **横断検索・類似事例提示**(「842件中63%」) | `vector_embeddings` 基盤はあるが case 用途は未構築 | ケースの埋め込み + 条件付き集計 + 根拠付き提示 |
| G6 | **3評価(本人/保護者/支援者)** | 本人(mynameis)+客観 | 保護者評価・支援者評価の構造化、Outcomeとの結合 |
| G7 | **保護者相談・本人の声の取込** | 保護者チャット等はあるが知識化していない | 相談・本人の声を L1→L2 へ載せる経路 |

補足: G4/G5 は**全国横断**(複数法人を越えた蒸留)を含むため、**法務4点(企画書§16-8)と匿名化**が前提。
横断学習は同意・地理範囲・保存年限の確定までゲートする(現状の同意モデルは施設内/個票連結まで)。

## 4. 実装フェーズ(D1〜D5)— 既存ロードマップへの追加

蒸留エンジンを既存 S系の上に段階追加する。各フェーズは S0〜S5 と同じ規律
(1コミット=1分類・同意ゲート・PII3層・k匿名・レビュー→デプロイ承認)で進める。

- **D1 構造化抽出 (G1)**: 連絡帳/支援案の保存・生成時に、本文から `事実/支援/結果/仮説/タグ` を抽出して
  `ai_revision_events` に付随する **構造化レコード**(またはjsonb列)へ保存。抽出はまずルール+軽量AI、PIIマスク前提。
  → L2 を埋める。S4 の program 分類と整合(タグ=実施プログラム/5領域)。
- **D2 問い返し・仮説提示 (G2)**: 記録入力に対しAIが**問い**と**因果仮説候補**を返すAPI/UI。
  「書かせる」から「考えさせる」へ。仮説候補は L2 の `仮説` を埋める入口。
- **D3 支援者成長モデル (G3)**: 作者ごとの記録特徴(事実のみ/仮説記述/支援設計)から **Lv1〜4 を判定**し、
  D2 の問い返し強度・添削・類似事例提示を**Lvで出し分け**。S4d レポートに成長指標として表示(成長志向)。
- **D4 支援知DB + 蒸留 (G4/G6)**: `Case/Observation/Intervention/Outcome/Hypothesis/Evaluation/Knowledge`
  スキーマを新設(L3/L4)。同意済みL2から**施設内**でまず蒸留(L3支援知)。3評価(本人/保護者/支援者)をOutcomeに結合。
- **D5 横断検索・根拠提示 (G5)**: ケース埋め込み + 条件(対象/状況)での類似検索と
  「類似N件中X%で有効」型の**根拠付き提示**。**全国横断は法務4点クリア後**にゲート解除(まず施設内→法人内→匿名化横断)。

## 5. 統合した今後の開発フロー(全体像)

### 完了(本番稼働)
- S0 統一スキーマ / S1 同意基盤+同意UI / S2・S3 配管(個別支援計画・モニタリング・職員アセスメント・連絡帳) /
  S4 多次元分析+実施プログラム分類 / S4c Layer2集計 / S4d 管理者レポート / S5 自己改善ループ(4文書)。

### 未完了(既存ロードマップ)
- **S6 成果(outcome)**: 能力評価スコアΔ / モニタリング達成度 / **mynameis×kiduri 主観×客観の一致**(§13)。
- **§11 修正理由UI**: 1クリックchips + 自由記述 → 動的カテゴリ昇格(修正理由を実データで埋め、S5の指示文を強化)。
- **S4e 特性の統制タグ**(分析次元の追加。要配慮)。

### 未完了(蒸留エンジン = 本書で追加)
- **D1 構造化抽出 → D2 問い返し・仮説提示 → D3 成長モデル → D4 支援知DB+蒸留 → D5 横断検索・根拠提示**。

### 推奨順(依存関係)
1. **§11 修正理由UI**(S5の質を底上げ。既存に直結、小)
2. **S6 成果(outcome)**(主観×客観。mynameis連携が既にある、中)
3. **D1 構造化抽出**(蒸留の入口。L2を埋める、中)
4. **D2 問い返し・仮説提示**(現場価値が高い、中)
5. **D3 成長モデル**(D1/D2の上、中)
6. **D4 支援知DB+蒸留 → D5 横断検索**(大。法務4点クリアが横断の前提)

## 6. 着手前に確認したい点

1. **横断(全国)学習の範囲と時期**: まず施設内蒸留→法人内→匿名化横断、の段階で良いか。法務4点(保存年限/地理範囲/同意UI/規約)の確定担当・期限。
2. **構造化抽出(D1)の方式**: ルール優先か、軽量AI併用か(PIIマスク前提)。タグは実施プログラム/5領域に揃えるか。
3. **問い返し(D2)の出し方**: 既存の「下書き生成」と併存か、置換か(現場の運用負荷)。
4. **成長モデル(D3)の用途**: 育成目的に限定(評価には使わない)を明文化するか。
5. **3評価(D4)**: 保護者評価・支援者評価の入力経路(既存チャット/評価表の活用)。
